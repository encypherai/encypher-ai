# Demo Walkthrough

## 5-Minute Live Demo

## Goal

Create the immediate reaction:

"WordPress should be able to do this."

## 1. Set the frame

Say:

"We think WordPress needs a provenance and authenticity layer for the AI era. This plugin is a working proof of what that can feel like in a normal WordPress publishing flow."

## 2. Show the install surface

Show:

- the Encypher dashboard WordPress integration card
- the download action
- the guided WordPress setup guide

What to say:

- the workflow starts with a standard plugin install
- the guide is designed to keep setup inside a familiar WordPress path
- the goal is low-friction activation for real publishing teams

## 3. Show the WordPress settings page

Show:

- Encypher → Settings
- secure email connect
- connection health
- launch readiness checklist

What to say:

- the recommended flow is email-based approval, not credential hunting
- the plugin can still support existing centralized credential management
- the setup experience is intentionally WordPress-native

## 4. Complete the connection flow

Show:

- entering a work email
- approving the secure connection
- returning to WordPress
- automatic connection completion

What to say:

- the first aha is that setup happens cleanly from inside WordPress
- the plugin provisions what it needs and confirms readiness
- this reduces the adoption friction that usually kills infrastructure tools

## 5. Publish a normal post

Show:

- a normal WordPress post in the editor
- publish or update action
- signing status in the plugin UI

What to say:

- this is not a separate workflow
- provenance becomes part of normal publishing
- edits preserve provenance instead of breaking it

## 6. Verify on the frontend

Show:

- a published post
- the verification badge
- provenance details visible to readers

What to say:

- this is where trust becomes visible
- provenance is not just hidden metadata
- readers and editors both get useful confirmation

## 7. Close with the platform question

Say:

"The plugin is the proof of workflow. The deeper question is what provenance should look like as an extensible capability for WordPress, especially as AI-assisted publishing becomes more common."

## 15-Minute Discussion Flow

If the conversation opens up, go here next:

1. Why provenance matters for WordPress now
2. What is standards-based vs provider-specific
3. Where extensibility points matter for the ecosystem
4. How AI-assisted authoring and agents could preserve provenance
5. What might belong in plugins vs broader WordPress abstractions

## Expected Questions

### "Does this require Encypher specifically?"

Answer:

"Today, Encypher provides the hosted signing and verification path in this implementation. But the broader architectural point is that WordPress should be able to support provenance workflows cleanly, with room for different implementation paths over time."

### "Why would this matter to core?"

Answer:

"Because provenance is becoming part of publishing itself, especially as AI-assisted workflows grow. Even if the full service layer remains plugin territory, WordPress can still benefit from having clean extensibility points and user-facing patterns for authenticity."

### "How does this relate to AI?"

Answer:

"AI makes provenance more important because authorship, transformation, and trust all get harder to reason about. A provenance layer gives WordPress a way to preserve trust even as the publishing stack becomes more automated."

### "What is the open-source story here?"

Answer:

"The strongest open-source story is around standards, interoperability, and extensibility. We want the workflow to fit the ecosystem, not force the ecosystem to fit one closed product model."
